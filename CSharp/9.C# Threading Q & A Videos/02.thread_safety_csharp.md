# Thread Safety in C# Multi-Threaded Environment

## 1. Introduction

In this video, we understand how to make an object **thread-safe** in a multi-threaded environment.

Before learning different techniques to make an object thread-safe, first we need to understand what **thread-safe** means.

---

## 2. What is Thread Safety?

Thread safety means an object should behave correctly even when multiple threads are accessing it at the same time.

If an object behaves abnormally when multiple threads access it concurrently, then that object is **not thread-safe**.

---

## 3. Example: Maths Class

Consider a simple class called `Maths`.

This class has two variables:

```csharp
int num1;
int num2;
```

It also has a method called `Divide()`.

Inside the `Divide()` method:

- A loop runs around `100,000` times.
- Random values are assigned to `num1` and `num2`.
- `num1` is divided by `num2`.
- After division, both variables are reset to `0`.

Example logic:

```csharp
class Maths
{
    int num1;
    int num2;

    public void Divide()
    {
        for (int i = 0; i < 100000; i++)
        {
            Random rnd = new Random();

            num1 = rnd.Next(1, 2);
            num2 = rnd.Next(1, 2);

            int result = num1 / num2;

            num1 = 0;
            num2 = 0;
        }
    }
}
```

---

## 4. Calling the Same Object from Multiple Threads

Now we create a `Maths` object and call the `Divide()` method from two different threads.

One call happens from the **main thread**, and another call happens from a **child thread**.

Example:

```csharp
class Program
{
    static Maths objMaths = new Maths();

    static void Main(string[] args)
    {
        Thread t1 = new Thread(objMaths.Divide);
        t1.Start();

        objMaths.Divide();
    }
}
```

Now two threads are accessing the same `objMaths` object at the same time.

---

## 5. Problem: Divide By Zero Exception

When we run the program, sometimes we may get this error:

```text
Attempted to divide by zero.
```

This happens because the object is not thread-safe.

---

## 6. Why Does This Error Happen?

Two threads are running the same `Divide()` method at the same time.

Assume this situation:

1. Main thread assigns random values to `num1` and `num2`.
2. Main thread is about to divide `num1 / num2`.
3. At the same time, child thread resets `num1` and `num2` to `0`.
4. Main thread continues and tries to divide by `0`.
5. This causes the divide by zero exception.

So the problem happens because shared variables are accessed and modified by multiple threads at the same time.

---

## 7. Solution: Thread Synchronization

To avoid this problem, we must ensure that only one thread can execute the critical section of code at a time.

The critical section is the part where:

- Random numbers are assigned.
- Division happens.
- Variables are reset.

In .NET, important synchronization techniques are:

1. `lock`
2. `Monitor`
3. `Mutex`
4. `Semaphore`

In this video, we focus mainly on `lock` and `Monitor`.

---

## 8. Using `lock` Keyword

The `lock` keyword ensures that only one thread can execute the code inside the lock block at a time.

Example:

```csharp
class Maths
{
    int num1;
    int num2;

    public void Divide()
    {
        for (int i = 0; i < 100000; i++)
        {
            lock (this)
            {
                Random rnd = new Random();

                num1 = rnd.Next(1, 2);
                num2 = rnd.Next(1, 2);

                int result = num1 / num2;

                num1 = 0;
                num2 = 0;
            }
        }
    }
}
```

Now, when one thread enters the `lock` block, another thread must wait until the first thread exits the block.

This prevents abnormal behavior and avoids the divide by zero exception.

---

## 9. Important Point About `lock`

Whatever statements are written inside the `lock` block can be executed by only one thread at a time.

So this code becomes thread-safe:

```csharp
lock (this)
{
    num1 = rnd.Next(1, 2);
    num2 = rnd.Next(1, 2);

    int result = num1 / num2;

    num1 = 0;
    num2 = 0;
}
```

---

## 10. Using `Monitor`

`lock` is actually a shortcut syntax for `Monitor`.

Both `lock` and `Monitor` are used for the same purpose: to allow only one thread to execute a block of code at a time.

Example using `Monitor`:

```csharp
class Maths
{
    int num1;
    int num2;

    public void Divide()
    {
        for (int i = 0; i < 100000; i++)
        {
            Monitor.Enter(this);

            Random rnd = new Random();

            num1 = rnd.Next(1, 2);
            num2 = rnd.Next(1, 2);

            int result = num1 / num2;

            num1 = 0;
            num2 = 0;

            Monitor.Exit(this);
        }
    }
}
```

---

## 11. Better Way to Use `Monitor`

When using `Monitor`, it is better to use `try-finally`.

This ensures that `Monitor.Exit()` is always called, even if an exception occurs.

```csharp
Monitor.Enter(this);
try
{
    Random rnd = new Random();

    num1 = rnd.Next(1, 2);
    num2 = rnd.Next(1, 2);

    int result = num1 / num2;

    num1 = 0;
    num2 = 0;
}
finally
{
    Monitor.Exit(this);
}
```

---

## 12. `lock` vs `Monitor`

| Feature | lock | Monitor |
|---|---|---|
| Syntax | Simple | More detailed |
| Internally uses | Monitor | Monitor |
| Easy to use | Yes | Slightly more complex |
| Recommended for simple cases | Yes | Can be used for advanced control |

`lock` is syntactic sugar over `Monitor`.

That means internally `lock` uses `Monitor.Enter()` and `Monitor.Exit()`.

---

## 13. Important Note

In real-time projects, it is better not to lock on `this`.

Instead, create a private lock object.

Better approach:

```csharp
class Maths
{
    private readonly object _lockObject = new object();

    int num1;
    int num2;

    public void Divide()
    {
        for (int i = 0; i < 100000; i++)
        {
            lock (_lockObject)
            {
                Random rnd = new Random();

                num1 = rnd.Next(1, 2);
                num2 = rnd.Next(1, 2);

                int result = num1 / num2;

                num1 = 0;
                num2 = 0;
            }
        }
    }
}
```

This is safer because outside code cannot lock the same object accidentally.

---

## 14. Summary

Thread safety means an object should work correctly when multiple threads access it at the same time.

In the example, two threads accessed the same `Maths` object and caused a divide by zero exception.

This happened because one thread was dividing while another thread reset the shared variables to `0`.

To solve this problem, we used synchronization.

The `lock` keyword ensures that only one thread can execute the critical section at a time.

`Monitor` also does the same thing, and `lock` is just a simpler syntax over `Monitor`.

For simple synchronization scenarios, `lock` is commonly used.

For advanced scenarios, we can use `Monitor`, `Mutex`, or `Semaphore`.

---

## 15. Key Interview Points

- Thread safety means safe access to shared data in a multi-threaded environment.
- Multiple threads accessing shared variables can cause race conditions.
- Race conditions can create unexpected errors like divide by zero.
- `lock` allows only one thread to execute a block of code at a time.
- `Monitor` is the underlying mechanism used by `lock`.
- `lock` is easier and cleaner for simple cases.
- Always keep the locked code as small as possible.
- Avoid locking on `this`; use a private readonly object instead.
