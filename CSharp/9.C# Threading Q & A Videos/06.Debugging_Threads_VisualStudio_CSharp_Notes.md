# Debugging Threads in C# Using Visual Studio - Interview Notes

## Overview

In multi-threaded applications, debugging can be confusing because multiple threads may execute the same method at the same time.

Visual Studio provides special debugging tools to inspect, pause, resume, and conditionally debug threads.

---

# Sample Scenario

Suppose we have a method called `MyMethod`.

```csharp
static void MyMethod()
{
    for (int i = 0; i < 10; i++)
    {
        i++;
        Thread.Sleep(5000);
    }
}
```

This method is called using two different threads.

```csharp
Thread o1 = new Thread(MyMethod);
Thread o2 = new Thread(MyMethod);

o1.Start();
o2.Start();

Console.ReadLine();
```

Now both threads execute the same method.

---

# Problem While Debugging

If we put a breakpoint inside `MyMethod`, Visual Studio stops execution.

But the issue is:

```text
Which thread is currently stopped at the breakpoint?
Thread o1 or Thread o2?
```

To identify this, we use the **Threads Window** in Visual Studio.

---

# Opening Threads Window in Visual Studio

Go to:

```text
Debug -> Windows -> Threads
```

This opens the Threads window.

---

# What Threads Window Shows

The Threads window shows:

- Main thread
- Worker threads
- Current debugging thread
- Thread location
- Thread name
- Thread status

---

# Main Thread

The main thread is the thread that runs:

```csharp
static void Main()
```

In Visual Studio, it is usually shown as:

```text
Main Thread
```

---

# Worker Threads

Worker threads are additional threads created by the application.

Example:

```csharp
Thread o1 = new Thread(MyMethod);
Thread o2 = new Thread(MyMethod);
```

These two become worker threads.

---

# Identifying Your Threads

In the Threads window, many threads may appear.

Some are created by:

- .NET runtime
- Visual Studio debugger
- System events
- Application framework

Your threads can usually be identified by their location.

Example:

```text
Location: MyMethod
```

---

# Importance of Naming Threads

By default, worker threads may appear as:

```text
<No Name>
```

This makes debugging difficult.

So it is a good practice to name your threads.

```csharp
Thread o1 = new Thread(MyMethod);
o1.Name = "ONE1";

Thread o2 = new Thread(MyMethod);
o2.Name = "TWO2";

o1.Start();
o2.Start();
```

Now Visual Studio clearly shows:

```text
ONE1
TWO2
```

---

# Current Debugging Thread

In the Threads window, Visual Studio shows a yellow arrow/sign for the currently active debugging thread.

This helps identify:

```text
Debugger is currently stopped on which thread?
```

Example:

```text
Current Thread: ONE1
```

---

# Important Debugging Behavior

When a breakpoint is hit:

```text
All threads are paused.
```

Even if the debugger is currently showing one thread, other threads are also paused at their respective execution points.

So if thread `ONE1` hits the breakpoint, thread `TWO2` is not running in the background. It is also paused.

---

# Switching Between Threads

To inspect another thread:

1. Open the Threads window.
2. Select the required thread.
3. Double-click on that thread.

Visual Studio will switch the debugging context to that thread.

Example:

```text
Double-click TWO2
```

Now you can see where `TWO2` is paused.

---

# Freeze and Thaw Threads

Sometimes we want to debug only one thread and pause the other threads.

Visual Studio provides:

- Freeze
- Thaw

---

## Freeze

Freeze pauses a selected thread.

Steps:

1. Open Threads window.
2. Right-click the thread.
3. Click **Freeze**.

Example:

```text
Freeze TWO2
```

Now `TWO2` will not run while debugging continues.

---

## Thaw

Thaw resumes a frozen thread.

Steps:

1. Right-click frozen thread.
2. Click **Thaw**.

Example:

```text
Thaw TWO2
```

Now `TWO2` starts running again.

---

# Why Freeze and Thaw Are Useful

Freeze and Thaw help in isolated debugging.

Example:

```text
Debug ONE1 only
Pause TWO2
```

This is useful when:

- Multiple threads are interfering
- Race conditions are happening
- You want to observe one thread clearly
- You want to reproduce thread-specific issues

---

# Conditional Breakpoints for Threads

Sometimes we want the breakpoint to stop only for a specific thread.

Example:

```text
Stop only when thread name is TWO2
```

This can be done using conditional breakpoints.

---

## Conditional Breakpoint Example

Right-click breakpoint and add condition:

```csharp
System.Threading.Thread.CurrentThread.Name == "TWO2"
```

Now the breakpoint will hit only when the current thread is `TWO2`.

---

# Benefits of Thread Naming

Thread naming helps in:

- Identifying threads in Visual Studio
- Conditional debugging
- Understanding execution flow
- Debugging race conditions
- Debugging deadlocks
- Debugging multi-threaded issues faster

---

# Complete Sample Code

```csharp
using System;
using System.Threading;

class Program
{
    static void Main(string[] args)
    {
        Thread o1 = new Thread(MyMethod);
        o1.Name = "ONE1";

        Thread o2 = new Thread(MyMethod);
        o2.Name = "TWO2";

        o1.Start();
        o2.Start();

        Console.ReadLine();
    }

    static void MyMethod()
    {
        for (int i = 0; i < 10; i++)
        {
            i++;
            Thread.Sleep(5000);
        }
    }
}
```

# Debugging Steps Summary

1. Put breakpoint inside threaded method.
2. Run application using F5.
3. Open:

```text
Debug -> Windows -> Threads
```

4. Identify current thread using yellow arrow.
5. Name your threads for easier debugging.
6. Double-click a thread to switch debugging context.
7. Use Freeze to pause a thread.
8. Use Thaw to resume a thread.
9. Use conditional breakpoint for specific thread debugging.


Thread naming helps identify which thread is executing during debugging.

```csharp
thread.Name = "WorkerThread1";
```

Freeze pauses a selected thread so it does not continue execution.

```csharp
System.Threading.Thread.CurrentThread.Name == "TWO2"
```
