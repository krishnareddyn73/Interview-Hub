
# Collection Interfaces in C# (.NET)

## Overview

.NET collections implement several important interfaces that promote:

- Encapsulation
- Polymorphism
- Loose coupling
- Better layered architecture (2-tier / 3-tier)

The four commonly discussed interfaces are:

- IEnumerable
- ICollection
- IList
- IDictionary

> Modern .NET also provides generic versions such as `IEnumerable<T>`, `ICollection<T>`, `IList<T>`, and `IDictionary<TKey,TValue>`, which are preferred for new applications.

---

# Why Use Collection Interfaces?

Instead of exposing a concrete collection like `ArrayList` or `List<T>`, expose an interface.

Benefits:

1. Restrict what the consumer can do (Encapsulation)
2. Swap collection implementations easily (Polymorphism)

---

# Example Architecture

```text
UI Layer
    │
    ▼
Business Layer
    │
    ▼
Collection (ArrayList / List<T>)
```

Expose an interface instead of the concrete collection to limit access.

---

# Encapsulation

Suppose the business layer returns an `ArrayList`.

```csharp
public ArrayList LoadCustomers()
{
    return customers;
}
```

The UI can call:

```csharp
customers.Add(new Customer());
customers.RemoveAt(0);
```

This may not be desirable if the UI should only read data.

---

# IEnumerable

`IEnumerable` is the most basic collection interface.

It allows:

- Enumeration (`foreach`)
- Read-only traversal

It does **not** expose methods like `Add()` or `Remove()`.

Example:

```csharp
public IEnumerable LoadCustomers()
{
    return customers;
}
```

Consumer:

```csharp
foreach (Customer customer in customers)
{
    Console.WriteLine(customer.CustomerName);
}
```

### Use When

- The caller only needs to iterate.
- You want maximum encapsulation.

---

# ICollection

`ICollection` extends `IEnumerable`.

Additional features:

- Count
- CopyTo()
- SyncRoot
- IsSynchronized

Example:

```csharp
public ICollection LoadCustomers()
{
    return customers;
}
```

```csharp
Console.WriteLine(customers.Count);
```

### Use When

- Enumeration is needed.
- Collection size is also required.

---

# IList

`IList` extends `ICollection`.

Additional features:

- Index-based access
- Insert()
- RemoveAt()
- IndexOf()
- Contains()

Example:

```csharp
IList list = new ArrayList();

list.Add("A");
list.Add("B");

Console.WriteLine(list[0]);
```

### Use When

- Sequential/index-based access is required.

---

# IDictionary

`IDictionary` represents key-value collections.

Implemented by:

- Hashtable
- Dictionary<TKey,TValue>

Example:

```csharp
IDictionary dictionary = new Hashtable();

dictionary.Add(1, "Shiv");

Console.WriteLine(dictionary[1]);
```

### Use When

- Key-value lookup is required.

---

# Interface Hierarchy

```text
IEnumerable
      │
      ▼
ICollection
      │
      ▼
IList

IDictionary
(separate hierarchy)
```

---

# Polymorphism

All collections implement interfaces.

Example:

```csharp
IEnumerable data;

data = new ArrayList();

data = new List<string>();

data = new Queue<string>();
```

The same interface variable can point to different collection implementations.

---

# Generic Interfaces (Recommended)

| Non-Generic | Generic |
|-------------|---------|
| IEnumerable | IEnumerable<T> |
| ICollection | ICollection<T> |
| IList | IList<T> |
| IDictionary | IDictionary<TKey,TValue> |

Benefits:

- Strong typing
- Better performance
- No casting
- No boxing/unboxing

---

# Which Interface Should You Return?

| Requirement | Interface |
|-------------|-----------|
| Only iterate | IEnumerable<T> |
| Iterate + Count | ICollection<T> |
| Index-based access | IList<T> |
| Key-value lookup | IDictionary<TKey,TValue> |

---

# Interview Questions

## Why expose interfaces instead of concrete collections?

To achieve encapsulation, loose coupling, and polymorphism.

---

## What does IEnumerable provide?

Iteration using `foreach`.

---

## What extra feature does ICollection provide?

`Count` and other collection-level operations.

---

## Which interface supports index-based access?

`IList` / `IList<T>`.

---

## Which interface represents key-value collections?

`IDictionary` / `IDictionary<TKey,TValue>`.

---

## Which interfaces are preferred today?

The generic interfaces from `System.Collections.Generic`.

---

# Key Takeaways

- `IEnumerable` → Read-only iteration.
- `ICollection` → Iteration + Count.
- `IList` → Index-based collection operations.
- `IDictionary` → Key-value collections.
- Exposing interfaces instead of concrete collections improves encapsulation and polymorphism.
- Prefer generic interfaces (`IEnumerable<T>`, `ICollection<T>`, `IList<T>`, `IDictionary<TKey,TValue>`) in modern .NET.


# Collection Interfaces - Encapsulation & Polymorphism (Part 2)

## Overview

This topic continues the discussion on collection interfaces and focuses on how they enable:

- Encapsulation
- Polymorphism
- Loose coupling
- Layered architecture (2-tier / 3-tier)

The key idea is that the **client depends on an interface, not on the concrete collection implementation**.

---

# Runtime Polymorphism

Suppose the Business Layer returns an `IEnumerable`.

The implementation can change internally without affecting the UI.

```text
Business Layer

Today
IEnumerable --> ArrayList

Tomorrow
IEnumerable --> Hashtable

Later
IEnumerable --> List<T>

Future
IEnumerable --> Queue<T>
```

The UI continues to work because it only knows about `IEnumerable`.

---

# Why Is This Useful?

The UI should not care whether the backend uses:

- ArrayList
- Hashtable
- List<T>
- Dictionary<TKey,TValue>
- Queue<T>
- Stack<T>

It only needs something that can be enumerated.

```csharp
IEnumerable customers = service.LoadCustomers();

foreach (var customer in customers)
{
    // Display customer
}
```

No changes are required in the UI if the backend collection changes.

---

# Encapsulation in Layered Architecture

Example:

```text
UI Layer
      │
      ▼
Business Layer
      │
      ▼
Repository
```

Instead of exposing:

```csharp
public ArrayList LoadCustomers()
```

Expose:

```csharp
public IEnumerable LoadCustomers()
```

The consumer can iterate but cannot modify the underlying collection.

---

# Read-Only Reporting Screen

A reporting screen generally needs only to display data.

Expose:

```csharp
IEnumerable<Customer>
```

The UI can:

- Iterate
- Display records

The UI cannot:

- Add items
- Remove items
- Modify the collection

---

# Data Entry Screen

If editing is required, expose a richer interface.

Examples:

```csharp
IList<Customer>
```

or

```csharp
ICollection<Customer>
```

These provide additional operations based on business requirements.

---

# Benefits of Using Interfaces

## 1. Loose Coupling

The UI depends on an abstraction instead of a concrete implementation.

---

## 2. Polymorphism

The same interface reference can point to different collection implementations.

```csharp
IEnumerable<Customer> customers;

customers = new List<Customer>();

customers = new Queue<Customer>();

customers = new Customer[] { };
```

---

## 3. Easier Maintenance

Changing the internal collection usually does not require changes to consuming code.

---

## 4. Better Encapsulation

Expose only the operations required by the client.

---

# Which Interface Should Be Returned?

| Requirement | Recommended Interface |
|-------------|----------------------|
| Read-only iteration | IEnumerable<T> |
| Iteration + Count | ICollection<T> |
| Index access | IList<T> |
| Key-value lookup | IDictionary<TKey,TValue> |

---

# Modern Best Practice

Prefer generic interfaces.

```csharp
IEnumerable<Customer>

ICollection<Customer>

IList<Customer>

IDictionary<int, Customer>
```

Avoid exposing concrete collections like:

```csharp
ArrayList

Hashtable

List<Customer>

Dictionary<int, Customer>
```

unless callers genuinely need implementation-specific features.

---

# Interview Questions

## Why expose IEnumerable instead of List<T>?

To hide implementation details, improve encapsulation, and reduce coupling.

---

## What OOP principles do collection interfaces support?

- Encapsulation
- Polymorphism
- Abstraction
- Loose Coupling

---

## Can the implementation change without affecting the UI?

Yes. As long as the returned interface contract remains the same.

---

# Key Takeaways

- Program against interfaces, not concrete collection classes.
- `IEnumerable<T>` is ideal for read-only data.
- Interface-based design enables runtime polymorphism.
- Clients should not know whether the backend uses `List<T>`, `Dictionary<TKey,TValue>`, or another collection.
- Choosing the right interface helps enforce business rules and keeps layered applications maintainable.
